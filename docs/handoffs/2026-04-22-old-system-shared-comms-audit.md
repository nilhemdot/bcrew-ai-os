# Old System Shared Communications Audit

Date: 2026-04-22

## What The Old System Was Actually Doing

The old system was not trying to replace Missive or replace Google meeting notes.

It was trying to build an intelligence layer on top of the places the company already works:

- Missive
- Gmail
- Slack
- Google meeting notes
- ClickUp

## Important Old-System Doctrine

Confirmed from old code and docs:

- `Missive` was the **primary** email intelligence surface because internal comments were visible there.
- `Gmail` was the fallback / broader mailbox layer.
- `Google delegated access` was the approved Google path.
- `Meeting notes` were already being generated by Gemini and then routed / filed.
- The real missing value was never source access alone.
- The real value was:
  - read
  - classify
  - extract
  - route
  - codify

## Most Important Old Skill

From `skills/bcrew-email-intelligence/SKILL.md`:

The old system explicitly scanned:

1. Missive inbox
2. Missive per-person conversations
3. Gmail across approved mailboxes
4. Calendar for conflict context
5. Strategy files / open loops / decision log

Then it tried to produce:

- decisions
- action items
- bottlenecks
- escalation-worthy items
- suggested assignee
- suggested action

That is the exact Foundation intelligence shape we still need.

## Most Important Meeting Skill

From `skills/bcrew-meeting-filer/SKILL.md`:

The old system already treated Gemini meeting notes as a real source.

It was doing:

1. search ai@ Drive for `Notes by Gemini`
2. transfer to secure Drive
3. parse attendees
4. classify restricted vs safe
5. file into the correct folder
6. record metadata

That means the rebuild should not treat meeting notes as a hypothetical future source.

## What The Rebuild Should Carry Forward

1. `Missive` is the shared inbox and comment layer.
2. `Gmail` is the broader mailbox layer.
3. `Slack` is another intelligence surface.
4. `Google Gemini meeting notes` are a real source.
5. The real missing layer is one governed pipeline that reads all of them and extracts:
   - decisions
   - tasks
   - blockers
   - follow-ups
   - routing suggestions

## Rebuild Implication

This work belongs in Foundation because it is shared business intelligence.

What belongs in future hubs:

- nudging a department
- auto-updating a department workflow
- coach behavior
- writing into department-specific systems without approval

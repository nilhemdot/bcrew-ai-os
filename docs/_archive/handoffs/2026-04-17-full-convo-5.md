## The Answers Locked In

### 1. Foundation vs runtime vs agent identity

These are separate layers.

- **Foundation** = strategy docs, source contracts, PostgreSQL operating memory, decisions, backlog, verification
- **OpenClaw** = live runtime layer for channels and agent execution
- **Harlan / Crewbert** = agent identities, not the runtime itself
- **Codex / Claude Code** = terminal-native coding and investigation tools

### 2. Harlan and Crewbert

- **Harlan** is Steve's personal assistant
- **Crewbert** is the BCrew system orchestrator
- the intended operating model is:
  - humans talk to personal assistants first
  - personal assistants delegate multi-step or cross-system work to Crewbert
  - Crewbert routes to specialist agents and tools

### 3. Terminal-native agents

Terminal-native agents are real leverage and are more powerful than thin prompt-only agents.

But the system should **not** make every agent terminal-native by default.

Correct split:

- terminal-native for coding, debugging, browser research, open-ended investigation
- constrained/tool-scoped for source sync, reporting, classification, daily briefs, structured business workflows

### 4. Agent homes

- Harlan should **eventually** live outside this repo
- repo-local coding agents are allowed to live inside a repo
- specialist business agents should be defined in the system registry/runtime, not as random persona folders

### 5. Memory

Memory is layered, not singular.

- Foundation/business memory -> PostgreSQL
- agent/runtime recall -> OpenClaw native memory baseline
- workspace/session identity -> repo/workspace files such as `AGENTS.md`, `SOUL.md`, `USER.md`
